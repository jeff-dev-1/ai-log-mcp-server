# PRD — ai-log-mcp-server（MVP）

> 服从 `CLAUDE.md`（L1）。本文件是产品需求层，只描述「为谁、解决什么、边界、怎么验收」。
> 具体 REST 端点清单 → 以 `WORKFLOW.md` 阶段 1 拉取的 `${APP_BASE_URL}/openapi.json` 为准，本文件**不列举、不臆造端点**。
> 技术性的 tool ↔ REST 端点映射 → 见 `DESIGN.md`，本文件只引用、不重画。

## 1. 目标用户与核心场景

| 角色 | 关心什么 | 核心场景 |
| --- | --- | --- |
| **SRE / 运维** | 故障定位、止血速度 | 线上告警后，在 Claude/Cursor 里用自然语言检索某时间窗、某服务的日志，快速圈定异常并定位影响范围 |
| **应用工程师** | 调试、发布回归 | 复现 bug 时按 trace/关键字检索相关日志；发布后对比错误率，确认是否引入回归 |
| **安全工程师** | 异常行为、审计追溯 | 排查可疑访问/越权时，按主体、时间、行为类型检索日志，做事件追溯与取证 |

共同点：**不切换工具**，直接在 MCP 客户端（Claude Desktop / Cursor）里通过对话调用平台日志分析能力。

## 2. MVP 功能边界

> 以下为**能力级**描述。能力到底由哪些端点实现，以阶段 1 拉取的 `/openapi.json` 为准。

### In Scope（MVP 必须具备的能力）

- **C1 日志检索**：按条件（如时间范围、服务/来源、关键字、级别）查询日志。
- **C2 单条/明细查看**：获取某条日志或某次结果的完整明细。
- **C3 触发分析**：调用平台已有的"AI 日志分析"能力（如异常识别/归因/摘要），由平台计算、MCP 透传结果。
- **C4 结果获取**：取回分析任务的产出。
- **C5 环境可切换**：通过 `APP_BASE_URL` 指向不同平台实例，能力不变。

> 上述能力**仅在 `/openapi.json` 暴露对应端点时**才落地为 tool；openapi 未提供的能力，本 MVP 不自造。

### Out of Scope（MVP 明确不做）

- ❌ 不重写/不复刻平台的分析算法、存储、聚合逻辑（一律调 REST）。
- ❌ 不做鉴权/登录（backend REST 开放，登录门只在前端）。
- ❌ 不做前端 UI、可视化图表、告警推送。
- ❌ 不做跨实例数据合并、缓存、长期持久化。
- ❌ 不暴露前端专用 / 登录相关端点（若 openapi 中存在）。
- ❌ 不新增除 stdio 外的传输方式。

## 3. 产品级数据流

```
用户(自然语言)
   │  在 Claude / Cursor 中提问
   ▼
MCP 客户端  ──选择 tool──►  ai-log-mcp-server (stdio)
                                  │  组装请求（不加业务逻辑）
                                  ▼
                          平台 REST API (${APP_BASE_URL})
                                  │  平台完成检索/分析
                                  ▼
                          响应 ──透传──► MCP 客户端 ──► 用户看到结果
```

- 用户用「能力」（§2 的 C1–C4），不需要知道端点。
- 「能力 → 具体 tool → REST 端点」的映射是技术细节，**见 `DESIGN.md`**，此处不重画。
- MCP server 在链路中只做"组装请求 → 调 REST → 透传"，不做计算。

## 4. 验收标准（可观察 / 可验证）

| # | 标准 | 验证方式 |
| --- | --- | --- |
| A1 | 每个 MVP 能力（C1–C4）至少对应 1 个可在 `tools/list` 中列出的 tool | 启动后执行 `tools/list`，逐能力核对存在对应 tool |
| A2 | 调用任一检索类 tool，返回的字段与 `/openapi.json` 中该端点的 response schema 一致（无多余/臆造字段） | 对比 tool 返回结构与 openapi 定义 |
| A3 | 触发分析类 tool 后，能取回平台产出，且产出来自平台 REST 响应（非本地生成） | 抓取请求确认转发到 `${APP_BASE_URL}`，比对返回 |
| A4 | 修改 `APP_BASE_URL` 后，请求目标地址随之改变，且无需改动代码 | 改环境变量后抓包/看日志确认目标 host 变化 |
| A5 | 调用过程中不出现任何登录/token 步骤 | 全链路无鉴权请求；代码内无 token 逻辑 |
| A6 | REST 返回非 2xx 时，tool 返回含 status code + 响应体的结构化错误，不静默吞错 | 构造一个失败请求，观察 tool 返回内容 |
| A7 | openapi 未暴露的能力不存在对应 tool | 比对 `tools/list` 与 openapi `paths`，无多出的 tool |

> 验收文案禁用"流畅""好用""快"等主观词；每条均为可执行的二元判定。

## 5. 风险与待确认问题

| # | 类型 | 描述 | 待确认 |
| --- | --- | --- | --- |
| R1 | 契约可达性 | 生成/运行环境可能直连不到内网 `192.168.88.210:8000` | 由谁、在何处拉取 `/openapi.json`？是否需要离线副本？ |
| R2 | 能力映射缺口 | §2 的 C1–C4 可能在 openapi 中无对应端点，或字段不足 | 阶段 1 拉取后逐能力核对；缺口如何处理（降级/反馈平台方）？ |
| R3 | 分析的同步性 | 分析可能是异步任务（提交→轮询→取结果） | openapi 是否区分"触发"与"取结果"两个端点？需不需要轮询能力？ |
| R4 | 参数语义 | 时间范围、分页、级别等参数的取值约定 openapi 未必说明 | 是否需要平台方补充字段语义文档？ |
| R5 | 数据敏感性 | 日志可能含敏感信息，经 MCP 客户端外发 | 是否有脱敏/字段过滤要求？（当前 Out of Scope，需确认风险接受度） |
| R6 | 开放 REST 的安全性 | backend 无鉴权，任何能到 `:8000` 的客户端均可调用 | 网络层是否已做隔离？是否可接受？ |
